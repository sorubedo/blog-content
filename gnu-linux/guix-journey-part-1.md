## 打脸来得太快

上个月我写了篇 [[gnu-linux/cloud-native-immutable-linux|云原生不可变 Linux 发行版，为什么是未来]]，论证不可变 + 容器化构建是桌面 Linux 的终极形态。那篇文章发出去没几天，[obedur-os](https://github.com/sorubedo/obedur-os) 的云端构建就炸了。

依赖冲突。某个包 A 要 `qt6-qtbase-gui-6.10.2-2.fc44.x86_64`，某个包 B 要 `qt6-qtbase-gui-6.11.1-1.fc44.x86_64`，dnf 直接罢工。构建在 GitHub Actions 上跑了 30 分钟，最后甩给我三个红色叉号。

> [!note] 并不是说不可变系统没用
> 客观讲，系统没有因此崩掉——构建失败了镜像根本没推上来，我也拉不到有问题的版本。退一万步说，就算拉下来了，一个 `rpm-ostree rollback` 就回去了。不可变系统**确实**保护了我。
>
> 但问题是：**保护归保护，问题还是没解决啊。**

依赖冲突不会因为你的根文件系统只读就自动化解，它只是把爆炸半径限制住了。你还是要面对那个该死的依赖版本纠葛，只是现在你面对它的地方从本地终端变成了 GitHub Actions 的日志页面。

这不是我想象中的"不可变未来"。想象中的未来是声明式配置一把梭，现实是你在 YAML 里和 rpm 包名较劲，然后祈祷 cloud build 别又报一串冲突。

所以我决定跑路。

## 我到底在讨厌什么

冷静下来想，让我不爽的不是不可变这个理念，是 Fedora Atomic 这套体系的**半吊子感**。

- 系统核心只读 ✓
- 原子化更新 ✓
- 但包管理层还是传统的 RPM 状态机 ✗

你用 `rpm-ostree install` 装包的时候，本质上还是在操作一个全局可变的状态空间——只不过这个状态被"分层"到了 ostree 的 overlay 上。装包顺序依然可能影响结果，依赖冲突依然是你要自己解决的事情。不可变性只是一个外挂的保险丝，底下的引擎还是那台老机器。

> [!warning] 真正的"依赖地狱"是什么
> 不是"系统炸了救不回来"——那个 Fedora Atomic 确实能救。是"我想要装一个包，但它和现有依赖冲突，而我不应该需要关心这件事"。
>
> 消费者不应该需要知道什么是 ABI 兼容性。这不叫"把用户当小白"，这叫正常的产品设计。

## macOS 把用户当小白，Windows 把用户当管理员，Linux 把用户当系统维护者

我最近重新想了那个经典对比：

- **macOS**：用户是小白。系统帮你做所有决定，你闭嘴用就行。想改？不给权限。
- **Windows**：用户是系统管理员（或者普通用户，看 UAC 弹没弹）。你可以改很多东西，但改出事了微软不会帮你擦屁股。
- **GNU/Linux**：用户是系统维护者。你不只是用户，你是 sysadmin。依赖冲突了？自己看日志。内核参数要调？自己来。想要某个新版软件但源里没有？自己编译。

我之前的文章说"桌面 Linux 需要消费者化"。现在我觉得——**其实不用**。

Android 已经是消费者化的 Linux 了，跑在 30 亿台设备上。ChromeOS 也是。嵌入式 Linux 到处都是。Linux 的"消费者化"早就完成了，只不过完成它的不是桌面发行版而已。

桌面 Linux 百花齐放，各发行版有自己的哲学，挺好的。**没必要把所有人都变成小白用户，也没必要把所有人变成系统管理员。** 选择权本身才是 Linux 的核心价值。你想当小白？用 Ubuntu LTS 装个 Flatpak 就够了。你想折腾？Arch wiki 在向你招手。你想搞声明式不可变？NixOS 和 Guix 已经在那了。

说到 Guix……

## 所以为什么是 Guix

其实说来说去，上面的反思都是给自己找台阶。真正的理由很简单：

> **Fedora Atomic 我玩腻了。**

我想试试新的东西。而 Guix 正好站在"不可变"和"声明式"这个维恩图的交汇点上：

- 像 NixOS 一样，**抛弃 FHS**，每个包有自己的独立路径（`/gnu/store/xxx-hash-xxx/`）
- 用 **Scheme (Guile)** 做配置语言，比 Nix 的纯函数式语言更"正常"一点（虽然 Scheme 对大多数人来说也不正常）
- **声明式系统配置**：你的整个系统是一个 `/etc/config.scm` 文件，重装系统 = 复制这个文件 + 跑 `guix system reconfigure`
- **真正没有依赖地狱**：两个包依赖不同版本的同一个库？各自找各自的 hash 路径，互不干扰
- **事务性升级**：升级出问题可以回滚到上一个 generation，和 Fedora Atomic 的 ostree rollback 异曲同工

> [!note] 为什么不是 NixOS
> NixOS 也很好，但 Nix 这个语言我是真的看着头疼。Guile Scheme 至少是正经的 Lisp 方言，学会了对以后写 Emacs 配置也有帮助（虽然我不太用 Emacs）。
>
> 另外 Guix 是 GNU 官方项目，只打包自由软件。这意味着很多非自由固件和驱动需要自己额外配置 channel。虽然麻烦了点，但至少这条线画得很清楚，不会稀里糊涂装了一堆专有 blob 不知道。
