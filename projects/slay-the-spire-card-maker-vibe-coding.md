## 起因：全网找不到一个好用的卡牌制作器

杀戮尖塔（Slay the Spire）打了快几百小时，想自己DIY几张卡牌图片乐呵乐呵。网上搜了一圈，两个可用选项：

1. **某在线工具**：满屏广告，必须注册登录才能下载导出——我就是做张卡牌，你要我邮箱干啥？
2. **另一个无广告的**：太原始了，全英文界面，而且 UI 长得像是 2010 年的产物。

说白了，这需求简单得很：开个画布、选卡牌底色、填名字、填描述、填费用、贴个图、导出 PNG。怎么就没人做个像样的呢？

> [!note] 碎碎念
> 既然没有，那就自己 vibe 一个。反正现在 AI 编程这么猛，uniapp 一把梭，跨平台全端通吃，完美。

## 技术选型：uniapp + DeepSeek v4 Pro

[uni-app](https://uniapp.dcloud.net.cn/) 是我一直想试试的框架——一套代码编译到小程序、H5、App，对于这种小工具再合适不过。模型选了 DeepSeek v4 Pro，毕竟便宜大碗，用起来也不心疼的。UI 组件库用了 [wot-design-uni](https://wot-design-uni.cn/)，暗色模式挺好看的，自带的表单组件省了不少事——当然，后面也给我挖了坑。

技术要点其实不复杂：

1. 卡牌模板用 Canvas 绘制——分层渲染卡牌底框、卡图、名称、费用、描述文字
2. 卡牌数据用 Vue 响应式绑定表单
3. 导出用 Canvas 的 `toDataURL` / `toTempFilePath` 转 PNG 保存

听起来简单，做起来……

## 离谱问题一：DeepSeek 的神级幻觉

卡牌制作器的核心是 Canvas 渲染。我让 DeepSeek 帮我写 Canvas 绘制逻辑，跑起来之后——

**页面一片空白——不对，倒也不是完全空白。** 整个页面上只显示了一行字：「连接服务器失败，请点击重试」。

我当时就愣住了。**我哪来的服务器？** 这纯前端工具，连个 `fetch` 都没写，哪来的「连接服务器失败」？以为是自己不小心引入了什么奇怪的网络请求，全局搜了一圈，零结果。

控制台没有任何报错。运行时日志干干净净。就只有 wot-ui 组件库丢出来的这一行字，像个无头冤案。

DeepSeek 开始「分析」了：

> 「这是 uni-app 的 Canvas API 在多平台下的兼容性问题。让我给h5平台加个包装......让我换个canvas库......让我把canvas推倒重来...... 」

然后它就顺着这个方向疯狂输出——换 API 写法、加 polyfill、检查平台编译配置、排查 manifest.json……一轮一轮地试，一轮一轮地失败。每轮都是几千上万 token 的推理。

**真相是什么？** `CardColor` 是一个 enum，在跨文件导入时类型信息丢了，传进 wot-ui 的表单组件后，组件内部解析失败触发了默认的错误状态——也就是那行「连接服务器失败，请点击重试」。wot-ui 的组件错误兜底文案写死了这句，不管你是真网络请求挂了还是数据格式不对，一概「服务器连接失败」伺候。

不是 Canvas API 的锅，不是平台兼容性，不是真的有服务器。就是一个 enum 类型在导入链上丢了，把 UI 组件库的状态机炸进了 error 分支。

> [!warning] 避坑警告
> AI 模型在调试「页面空白不加载」这类问题时很容易掉进幻觉陷阱。因为运行时没有报错，它只能猜测，而它的猜测往往倾向于「复杂的系统性问题」而非「简单的低级错误」。下次遇到类似情况，先别让 AI 发挥，**手动检查所有变量声明和数据流**。

这个问题来来回回耗了我估计大几千万 token。光是看它对着不存在的「Canvas 兼容性问题」长篇大论，我就想顺着网线爬过去关掉它的推理开关。

## 离谱问题二：vue-i18n，你算计我！

这个问题比上一个更阴间。

开发环境（`yarn dev:h5`）一切正常。所有文本组件完美显示，卡牌名称、费用、描述文字一字不落。

生产构建（`yarn build:h5`）之后——**某些文本组件直接消失了。** 不是样式隐藏，不是条件渲染，就是根本没有这个节点。

而且**完全没有报错**。`vue-i18n` 不给你任何警告，构建工具不给你任何提示，页面安安静静地少了几行字。

我排查了整整一个下午：

- 检查 `v-if` 条件 → 没问题
- 检查 CSS  → 没有
- 检查构建优化 tree-shaking → 不太像
- 检查组件生命周期 → 正常触发
- 怀疑是 uni-app 条件编译的锅 → 不是

最后终于锁定了 `vue-i18n`。

**原因：字符串里包含 `|` 和 `@` 这类特殊字符，vue-i18n 在编译时把它们当成了有特殊意义的符号处理，直接导致包含这些字符的文本节点在 production build 中被优化掉了。**

开发环境不触发是因为 dev 模式下 vue-i18n 走的是运行时解析，不会做这种激进的编译优化。一到 build，编译器看到 `|` 就觉得这是管道符 / pluralization 语法，看到 `@` 就觉得是 linked locale message，然后要么解析失败静默丢弃，要么直接 tree-shake 掉。

> [!warning] 避坑警告
> 用 vue-i18n 之前先读文档，尤其是关于**特殊字符转义**的部分。`|` 是 pluralization 分隔符，`@` 是 linked message 引用符。如果你的文本内容是游戏卡牌描述（比如「造成 `6|8` 点伤害」、「消耗 `@` 能量」），这些字符会被 i18n 编译器误解。解决方案要么用 `'@.lower:message'` 这种显式 linked syntax，要么把特殊字符用 Unicode 转义序列替代，要么干脆别用 vue-i18n，自己写个简单 i18n 方案。

这波确实是我菜——没读文档就直接上了 `vue-i18n`，默认它就是个翻译 key-value 映射，谁知道它有这么多暗坑。但话说回来，**一个 i18n 库在 build 时静默删除 DOM 节点而不给任何警告**，这设计是不是也太激进了？

## 其他零零碎碎的坑

还有一些小问题，不展开了。举一个吧——卡牌文字的视觉居中。Canvas 的 `textAlign: 'center'` 只是几何居中，视觉上 CJK 字符方块和英文字母的视觉重心完全不一样，不同字体的 baseline 偏移也不同。最后只能一个像素一个像素去试，调出一个"看着差不多"的值。结果手机上看感觉文字微微偏左，电脑上又还好——跨平台的像素级微调就是这么玄学。
